# 实体足球机器人 3v3 Demo 优化方案

## 1. Demo 理解总结

依据当前工程中的 `README.md`、`src/brain/config/config.yaml`、`src/brain/behavior_trees/*.xml` 和主要行为树代码，demo 大致结构如下。

| 模块 | 已确认内容 | 需要确认 |
|---|---|---|
| `vision` | 发布 `/booster_vision/detection`、`/booster_vision/line_segments`，用于识别球、机器人、球门柱、场地线等 | 模型在真实赛场光照下的识别稳定性 |
| `brain` | 核心决策节点 `brain_node`，使用 BehaviorTree 行为树控制比赛流程 | 不同机器人本地配置是否已分别设置 `player_id` / `player_role` |
| `game_controller` | 接收 RoboCup GameController UDP，发布 `/robocup/game_controller` | 3v3 比赛现场是否使用标准 GameController |
| 运动控制 | 通过 `LocoApiTopicReq` 等接口发送速度、头部、踢球/移动指令 | 底层步态参数、最大安全速度需要实体测试 |
| 行为树 | 有 `game.xml`、`demo.xml`、`chase.xml`、`shoot.xml`；正式比赛建议以 `game.xml` 为主 | 比赛时实际启动的是哪个 tree |

主要 ROS2 输入/输出：

| 类型 | 名称 |
|---|---|
| 订阅 | `/booster_vision/detection`、`/booster_vision/line_segments`、`/odometer_state`、`/low_state`、`/head_pose`、`/remote_controller_state`、`/robocup/game_controller` |
| 发布 | `/kick_ball`、`/play_sound`、`/speak`、`LocoApiTopicReq` |
| 参数 | `game.*`、`robot.*`、`strategy.*`、`obstacle_avoidance.*`、`locator.*`、`recovery.*`、`vision.*` |
| 状态来源 | GameController 状态、视觉球状态、里程计、定位状态、队友通信、倒地恢复状态 |

当前已支持的基础行为：

| 行为 | 当前支持情况 |
|---|---|
| 找球 | 已支持：`CamFindBall`、`CamFastScan`、`FindBall` |
| 跟踪球 | 已支持：`CamTrackBall` |
| 追球/带球 | 已支持：`Chase`、`SimpleChase`、低速 `Kick` 推球 |
| 调整射门角度 | 已支持：`Adjust`、`CalcKickDir` |
| 踢球/射门 | 已支持：`Kick`、`RLVisionKick` |
| 防守/守门 | 已支持：`GoalKeeperPlay`、`GoToGoalBlockingPosition` |
| 回位/回场 | 已支持：`GoToReadyPosition`、`GoBackInField` |
| 队友协作 | 已支持基础通信和 lead/assist 判断：`tmImLead`、`tmMyCostRank`、`Assist` |
| 比赛状态切换 | 已支持 `INITIAL`、`READY`、`SET`、`PLAY`、`END`、`FREE_KICK`、罚下状态 |
| 倒地恢复 | 已支持：`CheckAndStandUp` |
| 高级传接球 | 需要确认/后期再做 |
| 精确射门策略 | 部分支持，但很多区域参数仍偏硬编码，需要确认后再优化 |

## 2. 关键优化点优先级

| 优先级 | 优化点 | 原因 | 修改风险 | 预期收益 |
|---|---|---|---|---|
| 高 | 开启或调小近球限速：`limit_near_ball_speed`、`near_ball_speed_limit`、`near_ball_range` | 追球冲过球是实体机器人最常见问题 | 低 | 提高控球稳定性，减少撞球和摔倒 |
| 高 | 调整 `Chase` 的 `vx_limit`、`vy_limit`、`vtheta_limit`、`dist`、`safe_dist` | 前锋 chase 当前较激进，实体上可能过冲 | 低 | 追球更稳，球更容易留在脚前 |
| 高 | 调整 `Adjust` 的转向阈值和切向速度 | 射门前是否对准球门主要由 `Adjust` 决定 | 中 | 提高射门方向质量 |
| 高 | 明确 3 台机器人角色配置 | 当前通过 `player_role` 区分 striker / goal_keeper，3v3 需要避免全员抢球 | 低 | 立刻减少混乱和碰撞 |
| 高 | 第一阶段建立录像和数据记录规范 | 没有基线就无法判断调参是否有效 | 低 | 后续每次调参都有依据 |
| 中 | 调整 `kick_range`、`kick_theta_range`、`abort_kick_when_ball_moved` | 影响踢球触发时机和误踢概率 | 中 | 减少没对准就踢、球动了还踢 |
| 中 | 优化找球扫描节奏 | 找不到球时会头部扫描、原地转、回 ready 位 | 中 | 缩短丢球后的恢复时间 |
| 中 | 调整 `tm_ball_dist_threshold` 和 `ball_control_cost_threshold` | 决定队友球信息是否可信、谁主控 | 中 | 改善 3v3 协作 |
| 中 | 守门站位参数：`GoToGoalBlockingPosition`、`dist_to_goalline` | 防守机器人不能离门太远或太近 | 中 | 提升防守覆盖 |
| 中 | 避障参数：`avoid_during_chase`、`chase_ao_safe_dist` | 多机器人靠近球时容易互撞 | 中 | 降低碰撞和卡位 |
| 低 | 大力射门、绕人、复杂传球 | 当前部分参数存在但未完全参与决策 | 高 | 后期提升上限 |
| 低 | 球运动预测 | 配置里有预测参数，但当前是否完整生效需要确认 | 高 | 后期用于拦截和提前站位 |
| 低 | 自动视觉防守 `enable_auto_visual_defend` | 配置中提示不建议开启 | 高 | 暂不作为赛前主线 |

## 3. 3v3 基础策略

角色建议：

| 机器人 | 角色 | 行为原则 |
|---|---|---|
| 1 号 | 主攻 | 优先抢球、追球、调整、射门 |
| 2 号 | 辅助/接应 | 默认不抢第一脚，站在球后或侧后方，主攻丢球/倒地时补位 |
| 3 号 | 防守/守门倾向 | 默认守我方半场或球门前，只在球进入危险区时上抢 |

场景策略：

| 场景 | 建议策略 |
|---|---|
| 开球时 | 1 号慢速接近球，2 号在中场偏侧接应，3 号守门前；优先稳定轻推，不追求大力射门 |
| 找不到球时 | 先头部扫描，再小角度转身；不要三台一起冲场，3 号优先守位 |
| 我方机器人先看到球 | 看到球且控球成本最低者成为 lead，其余进入 assist/defense |
| 多个机器人同时看到球 | 用 `tmMyCostRank` 或距离球最近原则，只允许 1 台主抢 |
| 球在中场 | 1 号抢，2 号侧后跟随，3 号不越过中线太多 |
| 球靠近对方球门 | 1 号射门，2 号补射/防反，3 号保持中后场 |
| 球靠近我方球门 | 3 号优先封堵，1 号/2 号中最近者解围，另一台回防 |
| 球在边界附近 | 降速，先调整角度，把球往场内或对方半场方向踢 |
| 机器人丢失目标 | 保留短时间球记忆；超过记忆时间后回找球状态 |
| 动作执行失败 | 停止当前动作，重新找球或回位；倒地时先起身再定位 |

推荐状态机：

| 当前状态 | 切换条件 | 下一状态 |
|---|---|---|
| 找球 | 本机或队友球位置可信 | 接近球 |
| 接近球 | 距球小于追球阈值 | 对准球 |
| 对准球 | 球在可踢距离且角度满足 | 射门/解围 |
| 对准球 | 角度不满足 | 继续调整 |
| 射门 | 球移动过大或丢失 | 重新找球 |
| 射门 | 踢完 | 回位/继续追球 |
| 防守 | 球进入我方危险区 | 封堵/解围 |
| 防守 | 球远离我方半场 | 回位 |
| 避免重复抢球 | 自己不是 lead | 辅助/接应 |
| 恢复状态 | 倒地、定位丢失、长时间无球 | 起身、定位、找球 |

## 4. 分阶段路线图

| 阶段 | 目标 | 观察现象 | 调整内容 | 测试方法 | 成功标准 |
|---|---|---|---|---|---|
| 1. 跑通 demo 和基线 | 不急着改，记录默认表现 | 能否找球、追球、踢球、回位、起身 | 暂不改参数 | 单机和三机各跑 3-5 轮录像 | 明确最大短板 |
| 2. 单机基础调参 | 找球、追球、转向、射门前对准 | 是否冲过球、转向抖动、丢球 | 速度、近球限速、追球距离、角度阈值 | 直线球、斜向球、静止球测试 | 单机能稳定接近并踢到球 |
| 3. 单机比赛行为 | 处理边界球、丢球、射门角度 | 是否盲踢、球出界、卡边 | `Adjust`、`kick_range`、找球恢复 | 半场模拟攻门 | 射门前大多能对准 |
| 4. 3v3 角色分工 | 主攻/辅助/防守明确 | 是否多机抢球、互撞 | `player_id`、`player_role`、协作阈值 | 三机半场训练 | 同一时刻主抢一般只有 1 台 |
| 5. 攻防转换 | 根据球位置切换策略 | 回防是否慢、补位是否乱 | lead/assist 规则、防守触发线 | 完整小局对抗 | 失球后能快速回防 |
| 6. 赛前稳定性 | 降低摔倒和失控 | 快速转身、急停、碰撞 | 降速、限转、避障、起身恢复 | 多轮完整比赛 | 连续运行稳定，失误可恢复 |

## 5. 第一阶段详细执行清单

第一阶段原则：只观察，不调参；一次只改变测试场景，不改变代码和配置。

| 测试项目 | 观察内容 | 可能问题 | 记录方式 | 下一步判断 |
|---|---|---|---|---|
| 启动流程 | vision、brain、game_controller 是否正常启动 | 话题无数据、GameController 状态不切换 | 记录终端日志和启动命令 | 先解决通信/启动问题 |
| 视觉识别 | 球在远/近/边界是否能被看到 | 漏检、误检、光照影响 | 录像加 rerun/log | 若识别不稳，先调视觉/场地光照 |
| 找球 | 丢球后头部和身体如何搜索 | 扫描太慢、原地转过头 | 录像记录丢球到重新看到球时间 | 优先调扫描节奏 |
| 追球 | 接近球时是否减速 | 冲过球、撞球、左右抖 | 记录距离球 2m/1m/0.5m 行为 | 优先调近球限速 |
| 对准 | 踢球前是否绕到合适角度 | 直接乱踢、调整太久 | 俯拍录像 | 优先调 `Adjust` |
| 踢球 | 是否踢到球、方向是否稳定 | 空踢、球动了还继续踢 | 正面和侧面录像 | 调 `kick_range`/角度阈值 |
| 防守 | 3 号是否守住合理位置 | 离门太远、追球过深 | 半场防守录像 | 调守门站位 |
| 3v3 协作 | 是否三台同时抢球 | 碰撞、堵球、互相挡路 | 俯拍全场 | 调角色和 lead 规则 |
| 倒地恢复 | 摔倒后是否能起身并继续 | 起身失败、恢复后定位错 | 单独录像 | 先保证安全动作 |
| 边界球 | 球靠边时是否降速处理 | 把球带出界、机器人出界 | 边线测试 | 后续加边界策略 |

暂时不要动的参数：

| 类型 | 原因 |
|---|---|
| 大力射门相关参数 | 当前不是第一阶段核心，风险高 |
| 自动视觉防守 | 配置提示不建议开启 |
| 球预测参数 | 当前是否完整生效需要确认 |
| 复杂避障和绕人策略 | 先确认基础追球是否稳定 |
| 多个参数同时修改 | 会导致无法判断是哪一个参数产生效果 |

第一阶段必须记录的数据：

| 数据 | 建议记录方式 |
|---|---|
| 每次测试编号、日期、场地、机器人编号 | Obsidian 表格 |
| 启动命令和使用的行为树 | 直接复制命令 |
| 机器人角色配置 | 记录 `player_id`、`player_role` |
| 找球耗时 | 秒表/录像回看 |
| 追到球耗时 | 秒表/录像回看 |
| 是否冲过球 | 是/否/次数 |
| 是否踢到球 | 成功率 |
| 是否摔倒 | 次数和触发场景 |
| 是否多机抢球 | 次数 |
| 最明显短板 | 每轮只写 1-2 个 |

短板判断优先级：

| 现象 | 优先优化 |
|---|---|
| 经常看不到球 | 视觉/找球 |
| 能看到但追不上 | 追球速度和方向 |
| 追到后冲过球 | 近球限速 |
| 能控住但踢偏 | 对准和踢球触发 |
| 三机互撞 | 角色分工和协作 |
| 经常摔倒 | 全局速度、转向、急停 |
| 丢球后发呆 | 找球恢复状态 |

## 7. 下一步行动建议

下一步先做第一阶段基线测试：单机器人跑 `找球 -> 追球 -> 调整 -> 踢球`，再三机器人跑一小局。每轮都录像，先不要改参数；等确认最大短板是“看不见、追不稳、踢不准、协作乱、容易摔”中的哪一个，再进入第二阶段调参。
