# 策略逻辑分析

本文档总结 `k1_booster` 人形机器人足球比赛程序的整体策略逻辑，重点覆盖 `brain` 行为树、裁判机状态机、前锋/守门员策略、任意球策略、视觉感知、定位、避障、多机协作和动作执行链路。

## 1. 总体结论

这套程序是一个基于 ROS2 + BehaviorTree.CPP 的 RoboCup 人形机器人足球比赛系统。比赛策略核心集中在 `src/brain`：

- `Brain::tick()` 以约 100Hz 更新世界模型、裁判机状态、球记忆、出界状态、开球状态、多机协作状态等。
- `game.xml` 是正式比赛主行为树，根据裁判机状态和角色进入不同子树。
- 前锋使用 `StrikerPlay`：找球、追球、调整、视觉踢球、普通踢球、传中、辅助站位。
- 守门员使用 `GoalKeeperPlay`：守门站位、主动出击、追球、调整、解围、球门封堵。
- 任意球使用专门子树：根据我方/对方球权和机器人 cost rank 进行进攻/防守摆位。
- 视觉、裁判机、里程计、底层状态、手柄和队友通信共同构成策略输入。
- `RobotClient` 是动作输出层，最终把策略转换成底层行走、头部控制、踢球、起身、视觉踢球、守门封堵等 RPC 命令。

一句话概括：

> 程序通过视觉和裁判机建立实时比赛状态，用行为树按比赛阶段和角色选择动作，再通过底层运动接口执行追球、踢球、守门、防守、定位和多机协作。

## 2. 核心模块职责

| 模块 | 主要职责 |
|---|---|
| `src/brain` | 比赛策略核心：状态机、行为树、世界模型、裁判机处理、定位、协作、动作调用 |
| `src/vision` | 视觉识别：球、门柱、机器人、对手、场地标志点、场线、深度/地面平面、球运动预测 |
| `src/game_controller` | 接收 RoboCup GameController UDP 包并转换为 ROS2 裁判机消息 |
| `src/robocup_ros2_interface` | 定义视觉与裁判机消息接口 |
| `src/booster_ros2_interface` / `src/booster_msgs` | 定义底层机器人状态、手柄和 RPC 控制消息 |
| `src/ball_motion_predictor` | 球运动预测，主要被 vision 使用 |
| `scripts` / `launch` / `config` | 节点启动、配置加载、行为树选择、比赛参数设置 |

整体数据流如下：

```text
相机 / 深度 / 头部姿态
    -> vision_node
        -> /booster_vision/detection
        -> /booster_vision/line_segments
        -> /booster_vision/ball

RoboCup GameController UDP
    -> game_controller_node
        -> /robocup/game_controller

底层机器人状态
    -> /odometer_state
    -> /low_state
    -> /head_pose
    -> /remote_controller_state
    -> fall_down_recovery_state

brain_node
    -> 更新球、机器人、障碍物、定位、裁判机状态、队友通信
    -> BehaviorTree 决策
    -> RobotClient
        -> LocoApiTopicReq
        -> 行走 / 头部 / 踢球 / 起身 / 视觉踢球 / 守门封堵
```

## 4. Brain 主循环

`Brain` 是比赛大脑，核心对象包括：

```cpp
BrainConfig
BrainData
RobotClient
Locator
BrainTree
BrainCommunication
BrainLog
```

`main.cpp` 中约 100Hz 执行：

```cpp
brain->tick();
```

`Brain::tick()` 的逻辑可抽象为：

```text
Brain::tick()
    -> 输出 debug / log 信息
    -> 更新球、机器人、障碍物、开球等记忆
    -> 处理特殊比赛状态
         - 普通开球
         - 任意球开球
         - 得分
         - 罚时
         - 裁判机状态
    -> 处理多机协作
         - 计算自己控球成本
         - 进行队内排名
         - 判断自己是否 lead
         - 同步队友状态
    -> tick 行为树
```

设计思想是：

```text
Brain 维护世界状态
BehaviorTree 选择动作
RobotClient 执行动作
```

## 5. 运行时核心状态

运行时状态主要集中在 `BrainData`。

### 5.1 比赛状态

包括比分、判罚、是否开球、是否任意球开球、双方存活人数、真实二级比赛状态等：

```cpp
score
oppoScore
penalty[]
oppoPenalty[]
isKickingOff
isFreekickKickingOff
liveCount
oppoLiveCount
realGameSubState
```

这些状态影响：

- 是否能动。
- 是否等待对方开球。
- 是否进入任意球摆位。
- 是否被罚时停止。
- 是否执行进攻、防守或回位。

### 5.2 坐标与定位状态

```cpp
robotPoseToOdom
odomToField
robotPoseToField
```

球场坐标系约定：

```text
原点：球场中心
x 轴：指向对方球门
y 轴：指向机器人左侧
theta：逆时针为正
```

追球、踢球、守门、任意球站位、出界判断，都依赖 `field` 坐标下的机器人和球位置。

### 5.3 球状态

```cpp
ballDetected
ball
robotBallAngleToField
lose_ball
predictedBallPos
ballWillBreach
ballBreachPoint
ballInterceptPoint
```

含义：

- `ballDetected`：当前视觉是否看到球。
- `ball`：本机估计球位置。
- `tmBall`：队友报告球位置。
- `robotBallAngleToField`：机器人到球方向。
- `lose_ball`：视觉踢球退出等逻辑使用。
- `predictedBallPos`：预测球位置。
- `ballInterceptPoint`：守门拦截点。

### 5.4 感知对象

```cpp
robots
goalposts
markings
fieldLines
obstacles
```

用途：

| 对象 | 策略用途 |
|---|---|
| `robots` | 对手、障碍物、威胁判断 |
| `goalposts` | 射门方向、球门角度、自定位辅助 |
| `markings` | 自定位 |
| `fieldLines` | 自定位、出界判断 |
| `obstacles` | 避障、任意球摆位、追球绕行 |

### 5.5 踢球决策状态

```cpp
kickDir
kickType
isDirectShoot
```

| 变量 | 含义 |
|---|---|
| `kickDir` | 计划踢球方向，field 坐标系 |
| `kickType` | `shoot` / `cross` / `block` |
| `isDirectShoot` | 直接任意球时是否直接射门 |

### 5.6 多机协作状态

```cpp
tmStatus[]
tmMyCost
tmMyCostRank
tmImLead
tmReceivedCmd
tmImInVisualKick
shouldExitRLVisionKick
myStrikerIDRank
```

含义：

- 每台机器人计算自己控球成本。
- 成本低的机器人成为 lead。
- lead 负责控球/进攻。
- 非 lead 进入 assist 辅助站位。
- 如果队友更适合控球，本机可退出视觉踢球。

## 7. 正式比赛主状态机

正式比赛主行为树：

```text
src/brain/behavior_trees/game.xml
```

顶层结构：

```text
MainTree
    -> 默认 control_state = 3
    -> 手动 / 辅助分支
    -> pickup 后重新定位分支
    -> 自动比赛分支
        -> 自动起身
        -> 若被罚时：停止 + 定位/找球
        -> 若未罚时：
            -> TIMEOUT
            -> NORMAL
                -> INITIAL
                -> READY
                -> SET
                -> PLAY
                -> END
            -> FREE_KICK
                -> STOP
                -> GET_READY
                -> SET
```

### 7.1 手动 / 辅助分支

条件：

```xml
control_state==1 || assist_kick || go_manual || assist_chase
```

行为：

```text
assist_chase -> SimpleChase
assist_kick  -> Kick
```

主要用于调试、手柄辅助追球、辅助踢球和人工接管。

### 7.2 pickup 后重新定位

条件：

```xml
control_state==2
```

流程：

```text
未定位：CamScanField
已定位：CamFindAndTrackBall
然后：SelfLocateEnterField
```

用途是机器人被拿起或移出场后，从入场位置重新校准定位。

### 7.3 自动比赛分支

条件：

```xml
control_state==3 && !go_manual
```

流程：

```text
Speak auto
AutoGetUpAndLocate -> CheckAndStandUp

如果 gc_is_under_penalty：
    SetVelocity 停止
    未定位 -> CamScanField
    已定位 -> CamFindAndTrackBall
    SelfLocateEnterField

如果未罚时：
    按裁判机状态分支
```

## 8. 裁判机状态策略

### 8.1 TIMEOUT

条件：

```xml
gc_game_sub_state_type=='TIMEOUT'
```

行为：

```text
SetVelocity 停止
如果已定位，则继续 CamFindAndTrackBall
```

### 8.2 INITIAL

条件：

```xml
gc_game_state=='INITIAL'
```

行为：

```text
未定位：CamScanField
已定位：CamFindAndTrackBall
SelfLocateEnterField
```

含义：在场外入场位置站好，并完成入场前定位。

### 8.3 READY

条件：

```xml
gc_game_state=='READY'
```

行为：

```text
MoveHead pitch=0.35 yaw=0
GoToReadyPosition vx_limit=0.7
Locate
```

含义：进入场内准备位置，同时做自定位。

### 8.4 SET

条件：

```xml
gc_game_state=='SET'
```

行为：

```text
CamFindAndTrackBall
SetVelocity 停止
Locate
```

含义：站好等待比赛开始，只允许头部找球和定位，身体停止。

### 8.5 PLAY

条件：

```xml
gc_game_state=='PLAY'
```

角色分支：

```text
player_role == striker      -> StrikerPlay
player_role == goal_keeper  -> GoalKeeperPlay
```

### 8.6 END

条件：

```xml
gc_game_state=='END'
```

行为：

```text
SetVelocity 停止
```

## 9. 前锋正常比赛策略

文件：

```text
src/brain/behavior_trees/subtrees/subtree_striker_play.xml
```

整体流程：

```text
SelfLocate
Locate
    ↓
如果等待对方开球：
    CamFindAndTrackBall
    SetVelocity
    ↓
否则正常比赛：
    如果球出界：
        CamFindAndTrackBall
        GoBackInField
        Locate
    否则：
        CamFindAndTrackBall
        CalcKickDir
        StrikerDecide
        根据 decision 执行动作
```

### 9.1 等待对方开球

条件：

```xml
wait_for_opponent_kickoff
```

行为：

```text
CamFindAndTrackBall
SetVelocity
```

解除等待的条件大致是：

```text
球移动距离 > max(球距 * 0.15, 0.3m)
或
等待超过 10 秒
```

### 9.2 球出界处理

条件：

```xml
ball_out
```

行为：

```text
CamFindAndTrackBall
GoBackInField
Locate
```

含义：球出界时不盲目追出场，先回场内并继续定位。

### 9.3 前锋决策动作

前锋核心动作选择：

```xml
<CamFindAndTrackBall />
<CalcKickDir />
<StrikerDecide decision_out="{decision}" decision_in="{decision}" chase_threshold="1.0" />
<FindBall _while="decision=='find'" />
<Assist _while="decision == 'assist'" />
<Chase _while="decision == 'chase'" />
<RLVisionKick _while="decision == 'auto_visual_kick'" />
<Adjust _while="decision =='adjust'" />
<Kick _while="decision == 'kick'" />
<Kick _while="decision == 'cross'" />
```

`decision` 含义：

| decision | 动作 | 含义 |
|---|---|---|
| `find` | `FindBall` | 不知道球位置，找球 |
| `assist` | `Assist` | 自己不是主控球，走辅助位 |
| `chase` | `Chase` | 球较远，追球 |
| `auto_visual_kick` | `RLVisionKick` | 条件满足，底层视觉踢球接管 |
| `adjust` | `Adjust` | 球较近但角度不合适，绕球调整 |
| `kick` | `Kick` | 正常射门/踢球 |
| `cross` | `Kick` | 横传/传中/安全处理 |

### 9.4 Chase 追球逻辑

`Chase` 不是简单冲向球，而是根据踢球方向 `kickDir` 选择目标点：

```text
如果角度合适：
    target = 球后方 dist 距离的位置

如果角度不合适：
    target = 围绕球绕后的 circle_back 位置
```

追球过程中还会：

- 远球时保持大前进速度。
- 侧向球时加入横向切入。
- 近球时限制横向速度，避免跨过球。
- 若追球避障开启，目标方向有障碍时调用避障方向。

前锋 Chase 参数：

```xml
vx_limit="2.0"
vy_limit="0.75"
vtheta_limit="2.4"
dist="0.3"
safe_dist="0.65"
```

### 9.5 Adjust 调整逻辑

前锋接近球但角度不适合踢球时进入 `Adjust`。

参数：

```xml
turn_threshold="0.5"
range="0.4"
vx_limit="0.7"
vy_limit="0.6"
vtheta_limit="1.5"
vtheta_factor="1.5"
tangential_speed_far="0.7"
tangential_speed_near="0.15"
near_threshold="0.8"
no_turn_threshold="0.1"
turn_first_threshold="0.5"
```

作用：围绕球调整相对位置和朝向，调整完成后进入踢球或传中。

### 9.6 Kick 踢球逻辑

普通射门：

```xml
<Kick speed_limit="0.9" min_msec_kick="1000" msecs_stablize="1000"/>
```

传中/横传：

```xml
<Kick speed_limit="0.6" min_msec_kick="600"/>
```

如果 `abort_kick_when_ball_moved` 开启，踢球过程中球明显移动或丢失时可中止踢球。

### 9.7 自动视觉踢球 RLVisionKick

行为树节点：

```xml
<RLVisionKick min_msec_kick="1500" max_msec_kick="7000" range="6.0"/>
```

配置：

```yaml
enable_auto_visual_kick: true
auto_visual_kick_enable_dist_min: 0.35
auto_visual_kick_enable_dist_max: 2.5
auto_visual_kick_enable_angle: 0.55
```

含义：当球距离和角度满足条件时，Brain 调用底层视觉踢球技能，由底层接管靠近球和踢球动作。

## 10. 踢球方向策略 CalcKickDir

`CalcKickDir` 决定：

```cpp
BrainData::kickDir
BrainData::kickType
```

总体逻辑：

```text
读取球位置、机器人位置、球门位置、比赛状态
    ↓
计算球到对方左右门柱的可射门角度
    ↓
判断是否适合直接射门
    ↓
根据场景选择 shoot / cross / block
```

典型场景：

| 场景 | kickType | kickDir |
|---|---|---|
| 正常进攻且射门角度好 | `shoot` | 指向对方球门 |
| 射门角度差但球在前场 | `cross` | 横传到中路/门前 |
| 靠近己方门或防守状态 | `block` | 解围方向，远离己方球门 |
| 任意球开球 | `shoot` 或 `cross` | 根据球位置和直接射门判断 |
| 球靠近对方底线 | 继续向前处理 | 避免回踢 |

射门角度会考虑：

```cpp
goalPostMargin
```

即把门柱向内缩一点，减少打中门柱的概率。

## 11. 守门员正常比赛策略

文件：

```text
src/brain/behavior_trees/subtrees/subtree_goal_keeper_play.xml
```

整体流程：

```text
SelfLocate
Locate
    ↓
如果等待对方开球：
    CamFindAndTrackBall
    SetVelocity
    ↓
否则：
    如果球出界：
        CamFindAndTrackBall
        GoBackInField
        Locate
    否则：
        根据 goalie_mode 分支
```

守门员有两种模式：

```text
goalie_mode == attack
```

### 11.1 attack 模式

主动出击模式：

```xml
<CamFindAndTrackBall />
<GoalieDecide />
<GoToGoalBlockingPosition _while="decision=='find'" />
<GoToGoalBlockingPosition _while="decision == 'retreat'" />
<Chase _while="decision == 'chase'" />
<Adjust _while="decision == 'adjust'" />
<Kick _while="decision == 'kick'" />
```

守门员 `decision`：

| decision | 动作 | 含义 |
|---|---|---|
| `find` | `GoToGoalBlockingPosition` | 找不到球时回门线/封堵位 |
| `retreat` | `GoToGoalBlockingPosition` | 回防 |
| `chase` | `Chase` | 球可争抢时主动出击 |
| `adjust` | `Adjust` | 近球调整 |
| `kick` | `Kick` | 解围 |

守门员 Chase 参数更保守：

```xml
vx_limit="1.5"
vy_limit="0.3"
vtheta_limit="1.0"
dist="0.4"
safe_dist="0.6"
```

守门员解围踢球：

```xml
<Kick speed_limit="1.2" min_msec_kick="600"/>
```

### 11.2 guard 模式

纯守门模式：

```xml
<CamFindAndTrackBall />
<GoToReadyPosition _while="!ball_location_known" />
<GoToGoalBlockingPosition _while="ball_location_known" />
```

含义：

- 不知道球时回 ready 位置。
- 知道球时移动到球门封堵位置。
- 强调站位防守，不主动远距离追球。

## 12. 任意球策略

主树中任意球条件：

```xml
gc_game_sub_state_type=='FREE_KICK' && gc_game_state=='PLAY'
```

分为：

```text
STOP
GET_READY
SET
```

### 12.1 FREE_KICK STOP

行为：

```text
CamFindAndTrackBall
Locate if ball_location_known
SetVelocity
```

含义：规则停止阶段不能移动身体，但可以看球和定位。

### 12.2 FREE_KICK GET_READY

行为：

```text
Locate if ball_location_known
如果是前锋：StrikerFreekick
如果是守门员：GoalKeeperFreekick
```

### 12.3 前锋任意球

文件：

```text
subtree_striker_freekick.xml
```

我方开任意球：

```text
CamTrackBall if ball known
CalcKickDir
不知道球 -> FindBall
知道球 -> GoToFreekickPosition side="attack"
```

对方开任意球：

```text
CamTrackBall if ball known
CalcKickDir
不知道球 -> FindBall
知道球 -> GoToFreekickPosition side="defense"
```

`GoToFreekickPosition` 根据 `tmMyCostRank` 分工。

我方任意球 attack：

| rank | 站位 |
|---|---|
| 0 | 主罚，站在球后方 |
| 1 | 支援，站在球后更远位置 |
| 2 | 后场左侧防守位 |
| >=3 | 后场右侧防守位 |

对方任意球 defense：

| rank | 站位 |
|---|---|
| 0 | 最近防守/封堵点 |
| 1 | 第二防守点 |
| 2 | 后场左侧防守位 |
| >=3 | 后场右侧防守位 |

### 12.4 守门员任意球

文件：

```text
subtree_goal_keeper_freekick.xml
```

流程：

```text
CamFindAndTrackBall
如果不知道球：GoToReadyPosition
如果知道球：GoToGoalBlockingPosition
如果 odom_calibrated：SelfLocate trust_direction
```

### 12.5 FREE_KICK SET

行为：

```text
CamFindAndTrackBall
Locate if ball_location_known
SetVelocity
```

含义：已经摆好位置，身体停止，等待裁判机下一阶段。

## 13. 找球与头部策略

### 13.1 CamTrackBall

如果知道球位置：

- 看到球：根据球 bbox 中心和图像中心偏差调整头部 pitch/yaw。
- 没看到球但有本机记忆球：头转向记忆球位置。
- 没有本机球但有可靠队友球：头转向队友球位置。

像素容差：

```cpp
pixToleranceX = camPixX * 0.3
pixToleranceY = camPixY * 0.3
```

### 13.2 CamFindBall

找球逻辑：

```text
头部做类 Lissajous 扫描
身体以 vtheta = 1.5 rad/s 转动
```

关键参数：

```cpp
bodyVtheta = 1.5
headPeriod = 2.1
pitchCenter = 0.22
pitchAmp = 0.30
maxYawRate = 3.2
maxPitchRate = 2.2
acquireWindowMSec = 250
acquireCountThreshold = 2
```

看到球后短暂确认，然后停止身体旋转或进行轻微 reacquire。

### 13.3 CamScanField

用于定位扫描场地：头部在上下 pitch 和左右 yaw 间周期扫动，主要服务于 `INITIAL`、pickup 后重定位和 `SelfLocateEnterField`。

## 15. 球记忆、丢球、出界

### 15.1 球检测

`detectionsCallback()` 处理视觉检测：

- 过滤低置信度球。
- 排除明显不合理的球。
- 多个球中选择最可信球。
- 首次确认球位置需要短时间内多次检测，避免误检。
- 更新球在 robot 和 field 坐标中的位置。

配置：

```yaml
ball_confidence_threshold: 40.0
ball_memory_timeout: 3.0
```

### 15.2 丢球

如果长时间看不到球：

```text
ball_location_known = false
lose_ball = true
```

行为树会进入找球，或使用可靠队友球位置。

### 15.3 对方开球等待

对方开球或对方特殊球时，本机等待：

```text
球移动超过阈值
或
等待超过 10 秒
```

阈值：

```text
max(ball.range * 0.15, 0.3m)
```

### 15.4 球出界

`updateBallOut()` 和 `isBallOut()` 根据球的 field 坐标和场地边界判断球是否出界。球出界时：

```text
不追出界球
回场内
继续定位
```

行为树：

```xml
ball_out:
    CamFindAndTrackBall
    GoBackInField
    Locate
```

## 16. 多机协作策略

多机协作由以下逻辑共同完成：

```text
Brain::handleCooperation()
Brain::updateCostToKick()
BrainCommunication
Assist
RoleSwitchIfNeeded
```

核心思想：

```text
每台机器人计算自己控球成本
    ↓
通过队友通信交换状态
    ↓
成本最低者成为 lead
    ↓
lead 追球/踢球
    ↓
非 lead 进入 assist 站位
```

控球成本考虑因素：

| 因素 | 影响 |
|---|---|
| 是否知道球 | 不知道球增加成本 |
| 距离上次看到球的时间 | 越久成本越高 |
| 球距 | 越远成本越高 |
| 球方向偏差 | 偏角越大成本越高 |
| 障碍物 | 有障碍成本增加 |
| 队友是否更近 | 队友更近时成本增加 |
| 踢球方向调整难度 | 越难成本越高 |
| 是否跌倒 | 大幅增加成本 |
| odom 是否校准 | 未校准大幅增加成本 |

配置：

```yaml
enable_com: True
cooperation:
  enable_role_switch: true
  ball_control_cost_threshold: 5.0
tm_ball_dist_threshold: 1.5
```

## 17. 避障策略

障碍物来源：

1. 深度图点云。
2. 视觉识别到的对手/人。
3. 历史障碍物记忆。

深度图处理流程：

```text
采样深度图
    -> 转为机器人坐标系点
    -> 过滤地面、机器人自身区域、球附近点
    -> 栅格化
    -> 点数超过阈值生成 Obstacle
    -> 与历史障碍物融合
```

关键配置：

```yaml
obstacle_avoidance:
  depth_sample_step: 16
  obstacle_min_height: 0.3
  grid_size: 0.2
  max_x: 3.0
  max_y: 5.0
  exclusion_x: 0.4
  exclusion_y: 0.5
  occupancy_threshold: 20
  collision_threshold: 0.3
  safe_distance: 1.0
  avoid_secs: 2.0
  obstacle_memory_msecs: 500.0
  avoid_during_chase: true
  chase_ao_safe_dist: 1.5
  avoid_during_kick: false
  enable_freekick_avoid: true
```

追球时若目标方向障碍距离小于安全距离，会调用 `calcAvoidDir()` 选择绕行方向。

## 26. 策略特点总结

1. **裁判机驱动比赛阶段**：机器人严格根据 INITIAL、READY、SET、PLAY、END 和二级状态切换策略。
2. **行为树负责高层调度**：XML 决定流程，C++ 节点实现具体决策和动作。
3. **前锋以进攻为主**：找球、追球、绕球调整、射门、传中、视觉踢球。
4. **守门员以封堵和解围为主**：可在 guard/attack 间切换。
5. **对方开球遵守等待规则**：球移动或超时后才允许行动。
6. **球出界不盲追**：优先回场内并重新定位。
7. **任意球有专门摆位**：根据球权和 cost rank 分工。
8. **多机协作用 cost 决定 lead/assist**：降低多人同时抢球概率。
9. **视觉踢球可由底层接管**：满足距离和角度后进入 `RLVisionKick`。
10. **定位贯穿全流程**：READY、SET、PLAY、任意球、pickup 后都会持续定位。
11. **避障主要用于追球和任意球摆位**：基于深度图、视觉障碍物和历史记忆。
12. **动作出口统一**：所有动作最终通过 `RobotClient` 下发到底层 RPC。

---

## 27. 风险与调试关注点

1. **视觉球阈值影响找球能力**：vision 的 `Ball: 0.6` 可能比 brain 的 `ball_confidence_threshold: 40` 更关键。
2. **相机外参和头部姿态非常敏感**：会直接影响球距、踢球角度、自定位和出界判断。
3. **球运动预测会改变 Brain 看到的球位置**：`predict_time: 0.25` 过大可能导致近球过冲。
4. **GameController IP/白名单错误会导致状态机失真**。
5. **底层最小速度可能影响近球精细调整**。
6. **odom 未校准会显著影响任意球站位、守门位置、出界判断和协作 cost**。
7. **多机通信异常时，lead/assist 分工可能退化为单机逻辑或重复抢球**。
8. **障碍物误检可能导致追球绕行过多，漏检则可能撞人或被阻挡**。

---

## 28. 最终概括

这套比赛策略可以理解为：

```text
感知世界 -> 更新状态 -> 裁判机约束 -> 多机分工 -> 行为树决策 -> 底层动作执行
```

前锋的核心闭环是：

```text
找球 -> 算踢球方向 -> 追球 -> 绕球调整 -> 踢球 / 传中 / 视觉踢球
```

守门员的核心闭环是：

```text
找球 -> 判断威胁 -> 回防封堵 / 主动出击 -> 解围
```

任意球的核心闭环是：

```text
判断球权 -> 找球/定位 -> 按 cost rank 摆位 -> 等待 SET/PLAY
```

整体策略以规则状态机为骨架，以视觉和定位为输入，以行为树为调度，以 `RobotClient` 为动作出口，实现了完整的人形机器人足球比赛逻辑。
